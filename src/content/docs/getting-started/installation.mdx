HealthLog ships as a multi-arch Docker image (`linux/amd64` and `linux/arm64`) on [GitHub Container Registry](https://github.com/MBombeck/HealthLog/pkgs/container/healthlog). The bundled `docker-compose.yml` pulls it together with a PostgreSQL 16 service — no build step required for self-hosters.

## Prerequisites

- Docker Engine 24+ and Docker Compose v2+
- ~512 MB RAM and ~1 GB disk space
- A reverse proxy with HTTPS in front (Nginx Proxy Manager, Traefik, Caddy, …) — required for [Passkey authentication](/security/overview/) to work in production

## Quick Start (Docker Compose)

```bash
git clone https://github.com/MBombeck/HealthLog.git
cd HealthLog
cp .env.example .env
```

Generate the four required secrets and append them to `.env`:

```bash
echo "POSTGRES_PASSWORD=$(openssl rand -base64 24)" >> .env
echo "SESSION_SECRET=$(openssl rand -hex 32)"       >> .env
echo "ENCRYPTION_KEY=$(openssl rand -hex 32)"       >> .env
echo "API_TOKEN_HMAC_KEY=$(openssl rand -hex 32)"   >> .env
```

:::danger
The `ENCRYPTION_KEY` encrypts sensitive data in the database (Withings/AI tokens, VAPID keys, GitHub PATs). If it is lost or rotated without re-encrypting existing rows, that data becomes permanently unrecoverable. Back the value up somewhere safe.
:::

Bring the stack up:

```bash
docker compose up -d
```

Tail the logs to confirm a clean start:

```bash
docker compose logs -f app
# … once you see "Ready on http://0.0.0.0:3000", the app is up.
```

Open `http://localhost:3000` (or your reverse-proxy URL) and register the first account — it automatically gets admin rights. After that, disable open registration in **Settings → Administration**.

:::caution
Port 3000 should not be exposed to the public internet directly. Run a reverse proxy with HTTPS in front of the container (see [Self-Hosting → Reverse Proxy](/self-hosting/reverse-proxy/)). Passkeys (WebAuthn) only work in a secure context.
:::

## What's in the bundle

The repo's [`docker-compose.yml`](https://github.com/MBombeck/HealthLog/blob/main/docker-compose.yml) is the canonical setup. The minimum it needs to run looks like this:

```yaml title="docker-compose.yml"
services:
  app:
    image: ghcr.io/mbombeck/healthlog:latest
    container_name: healthlog-app
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: "postgresql://healthlog:${POSTGRES_PASSWORD}@db:5432/healthlog?schema=public"
      NODE_ENV: production
      SESSION_SECRET: "${SESSION_SECRET:?SESSION_SECRET is required}"
      ENCRYPTION_KEY: "${ENCRYPTION_KEY:?ENCRYPTION_KEY is required}"
      API_TOKEN_HMAC_KEY: "${API_TOKEN_HMAC_KEY:?API_TOKEN_HMAC_KEY is required}"
      NEXT_PUBLIC_APP_URL: "${NEXT_PUBLIC_APP_URL:-http://localhost:3000}"
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    container_name: healthlog-db
    environment:
      POSTGRES_USER: healthlog
      POSTGRES_PASSWORD: "${POSTGRES_PASSWORD:?POSTGRES_PASSWORD is required}"
      POSTGRES_DB: healthlog
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U healthlog"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: unless-stopped

volumes:
  pgdata:
```

The repo's version additionally includes a `build:` fallback so contributors can run `docker compose up --build` against their local source tree.

The [`.env.example`](https://github.com/MBombeck/HealthLog/blob/main/.env.example) in the repo documents every variable inline, including how to generate each secret. See also [Configuration → Environment Variables](/configuration/environment-variables/) for the long form.

## Pin a specific version

`:latest` follows `main`. For production, pin to a released tag:

```yaml
image: ghcr.io/mbombeck/healthlog:1.2.0
```

See the [container registry page](https://github.com/MBombeck/HealthLog/pkgs/container/healthlog) for available tags. Each image is built with [SLSA build provenance](https://slsa.dev/spec/v1.0/provenance) and an [SBOM](https://docs.docker.com/build/metadata/attestations/sbom/) attached — see the [Supply Chain section in `SECURITY.md`](https://github.com/MBombeck/HealthLog/blob/main/SECURITY.md#supply-chain) for verification commands.

## Build from source (contributors)

If you want to run a dev build instead of the published image:

```bash
docker compose up --build
```

This uses the `build:` block in the bundled `docker-compose.yml` and ignores the registry image. Or for a pure Node-based dev loop:

<Tabs>
<TabItem label="Local dev (pnpm)">

```bash
# Prereqs: Node 22+, pnpm 10+, PostgreSQL 16 reachable on localhost:5432
cp .env.example .env
# adjust DATABASE_URL hostname to "localhost"
pnpm install
pnpm db:generate
pnpm db:migrate:deploy
pnpm dev    # http://localhost:3000 with HMR
```

</TabItem>
<TabItem label="Production build">

```bash
pnpm install --frozen-lockfile
pnpm db:generate
pnpm build
pnpm db:migrate:deploy
pnpm start
```

</TabItem>
</Tabs>
