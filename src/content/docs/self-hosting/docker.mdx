If you're setting HealthLog up for the first time, start with [Getting Started → Installation](/getting-started/installation/) — that page has the canonical compose file and the Quick Start. This page documents the Docker-specific internals for operators who already have it running.

## Image source

Pre-built multi-arch images are published to GitHub Container Registry by the [`docker-publish.yml`](https://github.com/MBombeck/HealthLog/blob/main/.github/workflows/docker-publish.yml) workflow on every push to `main` and every `v*` tag.

| Property | Value |
| -------- | ----- |
| Registry | `ghcr.io/mbombeck/healthlog` |
| Architectures | `linux/amd64`, `linux/arm64` |
| Tags | `latest` (main), `vX.Y.Z`, `vX.Y`, `sha-<short>` |
| Base | `node:22-alpine` (multi-stage build) |
| User | `nextjs` (UID 1001, non-root) |
| Port | 3000 |
| Healthcheck | `GET /api/health` every 30 s |
| Provenance | [SLSA build attestation](https://slsa.dev/spec/v1.0/provenance) |
| SBOM | Attached to every image |

The non-root user means a container breakout cannot escalate to root on the host. See [`SECURITY.md`](https://github.com/MBombeck/HealthLog/blob/main/SECURITY.md#supply-chain) for the full supply-chain story including verification commands.

## Startup process

The image's `docker-entrypoint.sh` runs three steps before handing off to Next.js:

1. **Wait for PostgreSQL** — up to 30 connection attempts at 1 s intervals.
2. **Apply migrations** — `prisma migrate deploy`. Idempotent; safe to run on every restart.
3. **Start the server** — `node server.js`.

No manual migration step is required. If the database is unreachable after 30 seconds, the entrypoint exits non-zero and Docker restarts the container under the configured `restart: unless-stopped` policy.

## Updating to a new version

```bash
# Pin a tag in your docker-compose.yml first if you don't track :latest
docker compose pull
docker compose up -d
```

The new container starts, the entrypoint runs any pending migrations, then health-checks pass and the old container is replaced. See [Self-Hosting → Updates](/self-hosting/updates/) for the full guide including backup recommendations.

## Pinning versions

Production deployments should pin to a specific release:

```yaml
services:
  app:
    image: ghcr.io/mbombeck/healthlog:1.2.0 # not :latest
```

Each release is announced in [`CHANGELOG.md`](https://github.com/MBombeck/HealthLog/blob/main/CHANGELOG.md). Migrations are additive and tested in CI; downgrades are not generally supported.

## Reverse proxy

Port 3000 should not be exposed to the public internet directly. Run a reverse proxy with HTTPS termination in front of the container:

```yaml
services:
  app:
    image: ghcr.io/mbombeck/healthlog:latest
    # ...
    ports:
      - "127.0.0.1:3000:3000" # bind to loopback only when running behind a proxy on the same host
```

See [Self-Hosting → Reverse Proxy](/self-hosting/reverse-proxy/) for Caddy, Traefik, and Nginx Proxy Manager examples.

:::caution
Passkeys (WebAuthn) require a secure context. Without a valid TLS certificate on your reverse proxy, registration and login will not work.
:::
