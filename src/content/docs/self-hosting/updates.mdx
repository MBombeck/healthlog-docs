HealthLog is updated by pulling new Docker images. Database migrations run automatically on startup.

## Performing an Update

```bash
docker compose pull
docker compose up -d
```

The startup process runs `prisma migrate deploy` before starting the server. No manual migration steps are needed.

## Before Updating: Create a Backup

:::danger
**Always create a database backup before updating.** Database migrations may include schema changes that cannot be reversed. Without a backup, rolling back to a previous version is not possible.
:::

### Manual Backup

```bash
docker compose exec db pg_dump -U healthlog healthlog > backup_$(date +%Y%m%d_%H%M%S).sql
```

### Restoring a Backup

```bash
docker compose exec -T db psql -U healthlog healthlog < backup_20260308_120000.sql
```

## Automatic Backups

HealthLog creates a weekly automatic backup:

| Property | Value |
|----------|-------|
| Schedule | Sunday, 03:00 |
| Storage | Encrypted in the database |
| Managed by | pg-boss job |

Automatic backups can be viewed in the admin panel.

## Versioning

Docker images are published at `ghcr.io/mbombeck/healthlog`. Available tags:

| Tag | Description |
|-----|-------------|
| `latest` | Most recent stable release |
| `x.y.z` | Specific version (recommended for production) |

For maximum stability, pin a specific version tag:

```yaml
services:
  app:
    image: ghcr.io/mbombeck/healthlog:1.2.1
```

## Rollback

If an update causes problems:

1. Stop the containers:
   ```bash
   docker compose down
   ```

2. Restore the backup:
   ```bash
   docker compose up -d db
   docker compose exec -T db psql -U healthlog healthlog < backup_YYYYMMDD_HHMMSS.sql
   ```

3. Start with the previous image version:
   ```yaml
   image: ghcr.io/mbombeck/healthlog:1.2.0
   ```

   ```bash
   docker compose up -d
   ```

:::note
Rolling back to an older version may also require restoring a database backup if the newer version included schema changes.
:::
