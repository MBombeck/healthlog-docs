Security guidelines for running your own HealthLog instance.

## 1. Generate Cryptographic Keys

HealthLog requires three independent secrets. Each must be a 256-bit hex string:

```bash
# Session signing
openssl rand -hex 32  # -> SESSION_SECRET

# Encryption of sensitive DB fields (AES-256-GCM)
openssl rand -hex 32  # -> ENCRYPTION_KEY

# HMAC for API token hashing
openssl rand -hex 32  # -> API_TOKEN_HMAC_KEY
```

:::danger
The `ENCRYPTION_KEY` must **never** be changed or lost. It encrypts OpenAI keys, Telegram tokens, Withings tokens, notification config, and VAPID keys. If it is lost or changed, all encrypted data becomes permanently unreadable. Store this key separately (e.g. in a password manager).
:::

## 2. HTTPS Is Required

:::caution
HealthLog **requires HTTPS** in production. Session cookies are configured with `secure: true` and will not be sent over unencrypted connections. Passkeys (WebAuthn) only work over HTTPS or `localhost`.
:::

Use a reverse proxy for TLS termination. See [Reverse Proxy](/self-hosting/reverse-proxy/) for configuration examples.

## 3. Set APP_URL Correctly

The `APP_URL` environment variable is used for several security-critical features:

- **Passkey RP-ID** -- the Relying Party ID is derived from the domain
- **OAuth redirects** -- Withings OAuth callback URL
- **Webhook URLs** -- Withings notification endpoint

```bash
APP_URL=https://health.example.com
```

An incorrect value will cause passkey registration to fail and break OAuth flows.

## 4. Restrict Registration

After creating the admin account, you should disable public registration:

1. Sign in as admin
2. **Settings** -> **Administration** -> **Disable registration**

This prevents unauthorized users from creating accounts on your instance.

## 5. Change the Database Password

The `docker-compose.yml` ships with a placeholder password. Replace it before the first start:

```yaml
environment:
  POSTGRES_PASSWORD: a-secure-randomly-generated-password
```

The same password must be used in the app's `DATABASE_URL`:

```bash
DATABASE_URL=postgresql://healthlog:a-secure-randomly-generated-password@db:5432/healthlog
```

## 6. Keep Secrets Out of Version Control

Never commit your `.env` file to a Git repository. Make sure `.env` is listed in `.gitignore`. For CI/CD pipelines, use your provider's environment secrets feature.

## 7. Update Regularly

```bash
docker compose pull
docker compose up -d
```

Database migrations run automatically on startup. No manual intervention is needed. See [Updates](/self-hosting/updates/) for details.

## 8. Review the Audit Log

All security-relevant events are available in the admin panel under **Audit Log**:

- Failed login attempts (with IP and geo-location)
- Token creation and revocation
- Administrator actions
- Registrations

Check the audit log regularly for unusual activity.

## 9. Do Not Expose Port 3000 Publicly

The HealthLog container listens on port 3000. This port should **not** be directly reachable from the internet. Use a reverse proxy instead (see [Reverse Proxy](/self-hosting/reverse-proxy/)) and make sure a firewall blocks direct access to port 3000.

:::caution
Docker bypasses `ufw`/`iptables` firewall rules on Linux by default. If you map `ports: "3000:3000"` in your `docker-compose.yml`, the port may be publicly accessible despite your firewall. Either remove the port mapping and let the reverse proxy connect through the Docker network, or bind the port to localhost only: `ports: "127.0.0.1:3000:3000"`.
:::

## 10. Set Up External Database Backups

The automatic in-app backup (weekly, encrypted, stored in the database) is **not** a substitute for an external backup. Set up a regular PostgreSQL backup as well:

```bash
# Daily via cron
docker compose exec db pg_dump -U healthlog healthlog | gzip > /backups/healthlog_$(date +\%Y\%m\%d).sql.gz
```

If the entire database is lost, the in-app backups are gone too, since they are stored in the same database.
