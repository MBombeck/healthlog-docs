HealthLog uses a layered security model. This page provides an overview of all security-related mechanisms.

## Authentication

HealthLog supports two authentication methods:

### Passkeys (WebAuthn/FIDO2)

Passkeys are the recommended sign-in method. The implementation is based on **SimpleWebAuthn v13** and supports all common authenticator types (Touch ID, Face ID, Windows Hello, hardware keys like YubiKey).

- Fully passwordless registration and login
- Phishing-resistant through origin binding
- Multiple passkeys per account

### Passwords

Classic password authentication is available as a fallback:

| Property | Value |
|----------|-------|
| Hashing algorithm | Argon2id |
| Minimum length | 12 characters |
| Complexity | zxcvbn score >= 3 |

## Session Management

Sessions are stored server-side in PostgreSQL. The client receives a session cookie:

| Property | Value |
|----------|-------|
| Cookie name | `healthlog_session` |
| `httpOnly` | `true` |
| `secure` | `true` (production) |
| `sameSite` | `lax` |
| Sliding expiry | 30 days |

The session lifetime is automatically extended on each authenticated request. Expired sessions are periodically cleaned up from the database.

## Encryption of Sensitive Fields

Sensitive database fields are encrypted with **AES-256-GCM**. This applies to:

- OpenAI API keys
- Telegram bot tokens and chat IDs
- Withings OAuth tokens
- Notification configurations
- VAPID keys (Web Push)

The storage format is `base64(IV + AuthTag + Ciphertext)`. The key is provided via the `ENCRYPTION_KEY` environment variable.

:::danger
If the `ENCRYPTION_KEY` is lost, all encrypted fields become permanently unreadable.
:::

## API Token Security

Bearer tokens for the external API (`hlk_...`) are secured as follows:

- Stored in the database as **SHA-256 + HMAC** hashes
- The plaintext token is shown to the user **only once** at creation time
- Each token has **defined permissions** (scopes)
- Tokens can be **revoked** at any time

## Rate Limiting

HealthLog uses an in-memory sliding window for rate limiting:

| Endpoint | Limit |
|----------|-------|
| Login | 5 requests / 15 min |
| Registration | 5 requests / 15 min |
| Data ingest | 60 requests / min |
| Export | 10 requests / h |
| AI insights | 2 requests / h |
| Bug report | 3 requests / h |

When the limit is exceeded, the server responds with `429 Too Many Requests`. The `X-RateLimit-Remaining` and `X-RateLimit-Reset` headers indicate the current status.

## Security Headers

Every response includes the following security headers:

```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: [with nonces, no unsafe-inline]
```

The CSP uses dynamic nonces instead of `unsafe-inline` for script tags.

## Input Validation

All API inputs are validated using **Zod v4** schemas. Validation checks include:

- Data types and formats (e.g. email, UUID)
- Value ranges (e.g. systolic blood pressure 60-300 mmHg)
- String lengths and allowed characters
- Enum values

Invalid inputs are rejected with `422 Unprocessable Entity` and a descriptive error message.

## CSRF Protection

HealthLog protects against Cross-Site Request Forgery through three measures:

1. **SameSite=lax** cookie attribute prevents cross-origin requests
2. **JSON-only API** -- all mutations require `Content-Type: application/json`
3. **CSP form-action self** -- forms can only submit to the app's own domain

## Webhook Verification

Incoming webhooks (e.g. from Withings) are verified using **HMAC-SHA256**. Signature verification uses timing-safe comparison to prevent timing attacks.

## Audit Trail

All security-relevant actions are logged:

- Login attempts (successful and failed)
- Passkey registration and removal
- Token creation and revocation
- Administrator actions
- Account changes

Each entry includes the **IP address** and **geo-location** of the actor.

## No Telemetry or Tracking

HealthLog does not collect any usage data:

- Next.js telemetry is disabled (`NEXT_TELEMETRY_DISABLED=1`)
- No tracking code, no analytics, no third-party scripts
- External connections are only made when explicitly configured (OpenAI, Withings, Telegram)
