## Base URL

All API endpoints are relative to your instance's base URL:

```
https://your-instance.example.com/api
```

## Response Format

Every response follows a consistent envelope format:

```json
{
  "data": { ... },
  "error": null
}
```

On errors:

```json
{
  "data": null,
  "error": "Validation failed: password must be at least 12 characters"
}
```

Some endpoints also include a `meta` object with pagination data:

```json
{
  "data": [ ... ],
  "error": null,
  "meta": {
    "total": 142,
    "limit": 20,
    "offset": 0
  }
}
```

## Authentication

HealthLog supports two authentication methods:

| Method | Usage | Header/Cookie |
|--------|-------|---------------|
| Session Cookie | Browser clients | `healthlog_session` cookie (automatic) |
| Bearer Token | External API calls | `Authorization: Bearer hlk_...` |

Bearer tokens are created through the [token management](/api/external-ingest/) and are bound to specific permissions.

## Rate Limiting

Protected endpoints include rate limit headers in every response:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Remaining` | Remaining requests in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |

When the limit is exceeded, the API returns `429 Too Many Requests`.

## HTTP Status Codes

| Code | Meaning |
|------|---------|
| `200` | Success |
| `201` | Resource created |
| `204` | Success with no response body |
| `400` | Bad request |
| `401` | Not authenticated |
| `403` | Forbidden |
| `404` | Resource not found |
| `422` | Validation error (e.g. invalid field values) |
| `429` | Rate limit exceeded |
| `500` | Internal server error |

## Example Request

```bash
curl -s https://your-instance.example.com/api/auth/me \
  -H "Cookie: healthlog_session=abc123"
```

Response:

```json
{
  "data": {
    "id": "clx...",
    "username": "marc",
    "email": "marc@example.com",
    "role": "ADMIN"
  },
  "error": null
}
```
