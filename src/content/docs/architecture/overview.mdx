## Tech Stack

| Component | Technology |
|---|---|
| Framework | Next.js 16 (App Router, React Server Components) |
| Language | TypeScript (strict mode) |
| Database | PostgreSQL 16 |
| ORM | Prisma 7 with PrismaPg adapter |
| Background Jobs | pg-boss v12 |
| Styling | Tailwind CSS 4 + shadcn/ui (Dracula theme, dark-only) |
| Charts | Recharts 3 |
| Client State | TanStack Query 5 |
| Forms | React Hook Form 7 |
| Validation | Zod 4 |
| Auth | SimpleWebAuthn v13 (Passkeys) + session-based |
| PDF Export | jsPDF |

## Project Structure

```
src/
├── app/              # Next.js pages + API routes (App Router)
├── components/
│   ├── ui/          # shadcn/ui primitives (Button, Dialog, etc.)
│   ├── layout/      # Shell, navigation, header
│   └── [domain]/    # Feature-specific components (medications/, measurements/, etc.)
├── lib/
│   ├── auth/        # Session management, requireAuth(), requireAdmin()
│   ├── crypto.ts    # AES-256-GCM encryption/decryption
│   ├── db.ts        # Prisma client singleton
│   ├── jobs/        # pg-boss workers + job definitions
│   ├── logging/     # Wide event logging, Loki transport
│   └── [domain]/    # Feature libraries (withings/, telegram/, notifications/, etc.)

├── generated/       # Auto-generated (Prisma client)
├── i18n/            # Translations (de, en)
└── types/           # TypeScript definitions
```

## Request Flow

```
Client (Browser/PWA)
  → Middleware (proxy.ts)
    → API Route Handler
      → apiHandler() wrapper
        → Auth check (requireAuth / requireAdmin)
          → Business logic
            → Prisma (DB)
              → Response
```

### apiHandler Wrapper

All API routes use the `apiHandler()` wrapper. It handles:

- Consistent error handling
- Request validation (Zod schemas)
- Logging (wide events)
- Uniform response format

### Response Envelope

All API responses follow a consistent format:

```typescript
// Success
{ data: T, error: null }

// Error
{ data: null, error: string }

// Optional for lists
{ data: T[], error: null, meta: { total, limit, offset } }
```

## Authentication

HealthLog uses **Passkeys** (WebAuthn) as its primary authentication method, with an optional password fallback (Argon2id).

1. The client sends a request with a session cookie
2. Middleware checks for cookie presence
3. `requireAuth()` loads the session from the database and validates expiry
4. `requireAdmin()` additionally checks the user role (`role === "ADMIN"`)
5. For API token access: the token is hashed using SHA-256 + HMAC and verified against the database

## Logging

HealthLog uses **wide events** — structured log entries that bundle as much context as possible into a single event:

- Request metadata (method, path, duration, status)
- User context (userId, sessionId)
- Business context (varies by action)
- Optional: push to Loki for centralized logging

## Internationalization (i18n)

- Supported languages: **German** and **English**
- Configurable per user (`User.locale`)
- Global default via admin settings (`defaultLocale`)
- Translation files in `src/i18n/`
