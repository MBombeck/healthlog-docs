HealthLog uses **PostgreSQL 16** with **Prisma 7** as its ORM. The schema consists of 22 models, grouped into seven domains.

## Users and Auth

| Model | Description |
|---|---|
| `User` | User account with name, email, locale, admin flag, and settings |
| `Passkey` | WebAuthn credentials (public key, counter, transports) |
| `Session` | Active sessions with expiry and user reference |
| `AuthChallenge` | Temporary WebAuthn challenges (registration + login) |

A `User` can have multiple `Passkey`s and `Session`s. `AuthChallenge`s are deleted after use.

## Health Data

| Model | Description |
|---|---|
| `Measurement` | Individual reading with type, value, and timestamp |
| `MoodEntry` | Mood entry with level (1–5), optional comment, and tags |

### Measurement Types

| Type | Unit | Description |
|---|---|---|
| `WEIGHT` | kg | Body weight |
| `BLOOD_PRESSURE_SYS` | mmHg | Systolic blood pressure |
| `BLOOD_PRESSURE_DIA` | mmHg | Diastolic blood pressure |
| `PULSE` | bpm | Pulse rate |
| `BODY_FAT` | % | Body fat percentage |
| `SLEEP_DURATION` | h | Sleep duration |
| `ACTIVITY_STEPS` | steps | Daily step count |

Measurements always reference a `User` and carry a source (`source`): `MANUAL`, `WITHINGS`, or `IMPORT`.

## Medications

| Model | Description |
|---|---|
| `Medication` | Medication with name, dosage, and status (active/paused) |
| `MedicationSchedule` | Schedule: time window (windowStart/windowEnd), weekdays, optional dosage override |
| `MedicationIntakeEvent` | Individual intake (taken/skipped) with timestamp and source |
| `ReminderPhaseConfig` | Phase-based reminder escalation (GREEN, YELLOW, ORANGE, RED) |
| `TelegramReminderMessage` | Sent Telegram messages for later cleanup |
| `TelegramScheduledDeletion` | Scheduled deletion of Telegram messages |

A `Medication` has any number of `MedicationSchedule`s (e.g. morning and evening) and any number of `MedicationIntakeEvent`s. `ReminderPhaseConfig` defines per-medication escalation thresholds for the four reminder phases.

## Tokens

| Model | Description |
|---|---|
| `ApiToken` | External API access. The token value is stored as a **keyed HMAC-SHA-256 hash** (using the `API_TOKEN_HMAC_KEY` env var) — the plaintext is not retrievable after creation, and a DB dump alone does not let an attacker precompute matching hashes. |

## Integrations

| Model | Description |
|---|---|
| `WithingsConnection` | OAuth connection to Withings (access token, refresh token, user ID). Tokens are stored **encrypted with AES-256-GCM**. |

## Notifications

| Model | Description |
|---|---|
| `NotificationChannel` | Configured channel (Telegram, ntfy, Web Push) per user |
| `NotificationPreference` | Fine-grained control: which events go through which channel |
| `PushSubscription` | Web Push subscription data (endpoint, keys) |

## System

| Model | Description |
|---|---|
| `AppSettings` | Singleton with global app configuration (see [Admin Settings](/configuration/admin-settings/)) |
| `AuditLog` | Log of security-relevant actions (login, settings changes, token operations) |
| `DataBackup` | Metadata for encrypted database backups |
| `RateLimit` | Key-based rate limiting with counter and reset timestamp |
| `UserAchievement` | Unlocked achievements (gamification) with timestamp |

## Encryption

The following fields are stored encrypted with AES-256-GCM in the database:

- `User.openaiKeyEncrypted` (OpenAI API key)
- `User.telegramBotToken` (Telegram bot token)
- `User.withingsClientIdEncrypted`, `User.withingsClientSecretEncrypted`
- `User.moodLogUrlEncrypted`, `User.moodLogApiKeyEncrypted`
- `WithingsConnection.accessToken`, `WithingsConnection.refreshToken`
- `AppSettings.webPushVapidPrivateKeyEncrypted`, `AppSettings.githubIssueTokenEncrypted`
- `NotificationChannel.config` (depending on channel type)

The encryption key used is the `ENCRYPTION_KEY` environment variable.
