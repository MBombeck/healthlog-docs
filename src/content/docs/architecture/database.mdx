HealthLog verwendet **PostgreSQL 16** mit **Prisma 7** als ORM. Das Schema umfasst 22 Modelle, gruppiert in sieben Domaenen.

## User und Auth

| Modell | Beschreibung |
|---|---|
| `User` | Benutzerkonto mit Name, E-Mail, Locale, Admin-Flag, Einstellungen |
| `Passkey` | WebAuthn Credentials (Public Key, Counter, Transports) |
| `Session` | Aktive Sessions mit Ablaufzeit und User-Referenz |
| `AuthChallenge` | Temporaere WebAuthn Challenges (Registration + Login) |

Ein `User` kann mehrere `Passkey`s und `Session`s besitzen. `AuthChallenge`s werden nach Verwendung geloescht.

## Gesundheitsdaten

| Modell | Beschreibung |
|---|---|
| `Measurement` | Einzelner Messwert mit Typ, Wert und Zeitstempel |
| `MoodEntry` | Stimmungseintrag mit Level (1–5), optionalem Kommentar und Tags |

### Measurement Types

| Typ | Einheit | Beschreibung |
|---|---|---|
| `WEIGHT` | kg | Koerpergewicht |
| `BLOOD_PRESSURE_SYS` | mmHg | Systolischer Blutdruck |
| `BLOOD_PRESSURE_DIA` | mmHg | Diastolischer Blutdruck |
| `PULSE` | bpm | Pulsfrequenz |
| `BODY_FAT` | % | Koerperfettanteil |
| `SLEEP_DURATION` | h | Schlafdauer |
| `ACTIVITY_STEPS` | Schritte | Tagesschritte |

Messungen referenzieren immer einen `User` und tragen eine Quelle (`source`): `MANUAL`, `WITHINGS` oder `IMPORT`.

## Medikamente

| Modell | Beschreibung |
|---|---|
| `Medication` | Medikament mit Name, Dosis, Status (aktiv/pausiert) |
| `MedicationSchedule` | Zeitplan: Zeitfenster (windowStart/windowEnd), Wochentage, optionaler Dosis-Override |
| `MedicationIntakeEvent` | Einzelne Einnahme (genommen/uebersprungen) mit Zeitstempel und Quelle |
| `ReminderPhaseConfig` | Phasenbasierte Erinnerungs-Eskalation (GREEN, YELLOW, ORANGE, RED) |
| `TelegramReminderMessage` | Gesendete Telegram-Nachrichten fuer spaeteres Cleanup |
| `TelegramScheduledDeletion` | Geplante Loeschung von Telegram-Nachrichten |

Ein `Medication` hat beliebig viele `MedicationSchedule`s (z.B. morgens und abends) und beliebig viele `MedicationIntakeEvent`s. Die `ReminderPhaseConfig` definiert pro Medikament die Eskalationsschwellen fuer die vier Erinnerungsphasen.

## Tokens

| Modell | Beschreibung |
|---|---|
| `ApiToken` | Externer API-Zugang. Der Token-Wert wird als **SHA-256 Hash** gespeichert — der Klartext ist nach Erstellung nicht mehr abrufbar. |

## Integrationen

| Modell | Beschreibung |
|---|---|
| `WithingsConnection` | OAuth-Verbindung zu Withings (Access Token, Refresh Token, User-ID). Tokens werden **AES-256-GCM verschluesselt** gespeichert. |

## Benachrichtigungen

| Modell | Beschreibung |
|---|---|
| `NotificationChannel` | Konfigurierter Kanal (Telegram, ntfy, Web Push) pro User |
| `NotificationPreference` | Feinsteuerung: welche Events ueber welchen Kanal |
| `PushSubscription` | Web Push Subscription-Daten (Endpoint, Keys) |

## System

| Modell | Beschreibung |
|---|---|
| `AppSettings` | Singleton mit globaler App-Konfiguration (siehe [Admin-Einstellungen](/configuration/admin-settings/)) |
| `AuditLog` | Protokollierung sicherheitsrelevanter Aktionen (Login, Settings-Aenderungen, Token-Operationen) |
| `DataBackup` | Metadaten zu verschluesselten Datenbank-Backups |
| `RateLimit` | Key-basiertes Rate Limiting mit Zaehler und Reset-Zeitpunkt |
| `UserAchievement` | Freigeschaltete Achievements (Gamification) mit Zeitstempel |

## Verschluesselung

Folgende Felder werden mit AES-256-GCM verschluesselt in der Datenbank gespeichert:

- `User.openaiKeyEncrypted` (OpenAI API Key)
- `User.telegramBotToken` (Telegram Bot Token)
- `User.withingsClientIdEncrypted`, `User.withingsClientSecretEncrypted`
- `User.moodLogUrlEncrypted`, `User.moodLogApiKeyEncrypted`
- `WithingsConnection.accessToken`, `WithingsConnection.refreshToken`
- `AppSettings.webPushVapidPrivateKeyEncrypted`, `AppSettings.githubIssueTokenEncrypted`
- `NotificationChannel.config` (je nach Kanal-Typ)

Der verwendete Schluessel ist die Umgebungsvariable `ENCRYPTION_KEY`.
